## 1. なぜこのアプリで勝てるか

### アプリが解決する問題

#### 「スーパーに行くと、決めていなかったものをついつい買ってしまう」

買い物での衝動買いや食材の買いすぎは、多くの家庭が抱える悩みです。このアプリはその問題を「献立を先に決める → 決めたものだけ買う」という仕組みで解決します。

```
献立を週単位で決める
      ↓
必要な食材が買い物リストに自動で出る
      ↓
リストにあるものだけを買う
      ↓
無駄遣いがなくなる・食費が下がる
```

### ターゲットユーザー

| ターゲット | 抱えている悩み |
| --------- | ----------- |
| 食費を節約したい主婦・共働き世帯 | 買い物のたびに余計なものを買ってしまう |
| 一人暮らしで食材を腐らせがちな人 | 使い切れない食材が出る・献立が思いつかない |
| 家計管理を改善したい人 | 食費がどこに消えているかわからない |

日本国内の対象世帯：推定 **3,000万世帯以上**

### 競合との差別化

既存のレシピアプリは「何を作るか」を提案するが、本アプリは**「何を買うかを事前に決める」**ことに特化しています。

| 比較軸 | レシピアプリ（クックパッド等） | 節約・家計アプリ | 本アプリ |
| ----- | ------------------------ | ------------ | ------- |
| 核心的な価値 | 料理レシピの提供 | 支出の記録・分析 | **買い物前に計画を決める** |
| 献立の週単位管理 | なし〜弱い | なし | ◎ |
| 買い物リスト自動生成 | 一部あり | なし | ◎ 献立から自動集計 |
| シンプルさ | 機能過多 | 機能過多 | ◎ 3タブのみ |
| オフライン動作 | △ | △ | ◎ 完全ローカル |
| 料金 | 無料〜月額 | 無料〜月額 | 無料（広告除去のみ課金） |

### 強み

- **アプリの目的がシンプル**：「節約のために計画買いをする」という一点に絞られており、ユーザーに伝わりやすい
- **MVP は完成している**：週単位の献立管理・買い物リスト自動生成・料理履歴の3機能が動作済み
- **React Native / Expo**：iOS / Android 同時対応が容易
- **バックエンドなし**：AsyncStorage のみのため運営コストほぼゼロ

## 2.5 課金システムの選択肢

モバイルアプリで収益を得る手段はいくつかあります。本アプリでの採用可否も整理します。

### 課金モデルの種類

| 種類 | 仕組み | 本アプリへの適用 |
|-----|-------|---------------|
| **IAP 買い切り** | アプリ内で一度だけ支払う（広告除去・機能アンロックなど） | ★ 主軸（広告除去 ¥300） |
| IAP サブスクリプション | 月額・年額の継続課金 | 将来的な拡張で検討 |
| アプリ有料販売 | App Store でダウンロード自体を有料にする | 不採用（DL 数が稼ぎにくい） |
| **広告収入** | 無料ユーザーに広告を表示して収益化（AdMob など） | ★ サブ収益源 |
| Web 課金（Stripe など） | アプリ外の Web サイトで決済する | 審査が厳しく今回は対象外 |

### プラットフォームの IAP 仕組み

iOS と Android では、課金処理は必ず各ストアの決済基盤を通す必要があります。

| プラットフォーム | 決済基盤 | 手数料 | Expo での実装 |
|--------------|--------|------|-------------|
| iOS (App Store) | Apple In-App Purchase | **30%**（年間売上 ¥100万以下は 15%） | `expo-in-app-purchases` |
| Android (Google Play) | Google Play Billing | **15%** | 同上 |

> **ポイント**：売上の 15〜30% はプラットフォームに取られる。ユーザーが ¥300 払っても手取りは ¥210〜255。

### 課金管理サービス：RevenueCat（推奨）

iOS / Android 両方の IAP を一元管理できる外部サービス。

| 項目 | 内容 |
|-----|------|
| 何ができるか | iOS・Android の購入状態を統合管理、購入復元、ダッシュボードで収益分析 |
| 料金 | 月間収益 **$2,500（約38万円）以下は無料** |
| メリット | プラットフォームごとの実装差異を吸収。コードが大幅に簡単になる |
| デメリット | 外部サービスへの依存。将来的に有料化の可能性あり |

### 本アプリの推奨構成（まとめ）

| レイヤー | 採用技術 | 理由 |
|---------|--------|------|
| iOS / Android IAP | Apple IAP + Google Play Billing（RevenueCat 経由） | 両 OS を一元管理できる・無料枠で十分 |
| 広告 | Google AdMob | 無料ユーザーからの安定した収益源 |
| 課金管理・分析 | RevenueCat（無料プラン） | 購入復元・収益ダッシュボードが容易に構築できる |

### Step 3.5：RevenueCat に iOS アプリを登録して本番 API キーを取得する

#### 現在の状態

RevenueCat に「Test Store」のみ登録されており SDK キーは `test_` プレフィックスのみ。iOS 本番用の `appl_` キーを得るには、Apple App Store アプリを RevenueCat に追加する必要がある。

#### キーの使い分け

| フェーズ | 使用キー |
|---------|---------|
| Sandbox テスト | `test_` のままで OK（動作する） |
| TestFlight / App Store 提出前 | `appl_` キーに差し替える |

#### RevenueCat に iOS アプリを追加する手順

1. app.revenuecat.com にログイン
2. 左メニュー「Kondate Cart」プロジェクトを選択
3. 左メニュー「**Apps**」を選択
4. 右上「**+ New app**」をクリック
5. 以下を入力して保存：
   - App name：`Kondate Cart`
   - App store：**Apple App Store** を選択
   - Bundle ID：`com.nakakei6439.kondatecart`
   - App Store Connect App-Specific Shared Secret：下記手順で取得
6. 保存後「API Keys」ページに `appl_` で始まるキーが出現する
7. `src/store/purchaseStore.ts` の `REVENUECAT_API_KEY` を `appl_` キーに差し替える

#### App Store Connect Shared Secret の取得方法（手順5の補足）

1. appstoreconnect.apple.com にログイン → 対象アプリ（先に Step 6 でアプリ登録が必要）
2. 「App 情報」→「App-Specific Shared Secret」→「Shared Secret を管理」→「新規生成」
3. 生成されたキーをコピーして RevenueCat の入力欄に貼り付ける

> App Store Connect にアプリ登録がまだの場合は Step 6 を先に行ってから戻ること。

### Step 3.6：Sandbox テストで課金フローを確認する

#### 前提条件

#### 1. Sandbox テスト用 Apple ID を作成する

3. 「＋」ボタンで新規 Sandbox アカウントを作成
   - メールアドレス：専用の未使用メール（例：sandbox.test.xxx@gmail.com）
   - パスワード：任意（後で使うので控えておく）
4. 「保存」

#### 2. 実機を Sandbox モードに切り替える

1. iPhone の「設定」アプリを開く
2. 「App Store」→「SANDBOX アカウント」→「サインイン」
3. 先ほど作成した Sandbox 用 Apple ID でサインイン
   （通常の Apple ID とは別管理。サインアウト不要）

#### 3. アプリで購入テストを行う

1. Development Build を起動
2. 「買い物リスト」タブ → 右上の「⚙」ボタンで設定を開く
3. 「広告を消す ¥300」ボタンをタップ
4. Sandbox の確認ダイアログが表示される（「[SANDBOX] このアプリ内購入を確認」）
5. Sandbox アカウントのパスワードを入力して「購入」
6. 購入後に広告が非表示になることを確認

#### 4. 購入復元のテスト

1. アプリをアンインストール → 再インストール
2. 設定を開く → 「購入を復元」をタップ
3. 広告が再び非表示になることを確認

#### 5. よくある問題

| 症状 | 原因 | 対処 |
|------|------|------|
| 「商品が見つかりません」と出る | App Store Connect の商品ステータスが「承認済み」になっていない | App Store Connect で商品ステータスを確認 |
| Sandbox ダイアログが出ない | RevenueCat の初期化が失敗している | コンソールログを確認（`Purchases.setLogLevel(LOG_LEVEL.DEBUG)` が有効か確認） |
| 購入後も広告が出る | エンタイトルメント ID のミスマッチ | RevenueCat ダッシュボードの ID と `purchaseStore.ts` の `ENTITLEMENT_ID` を照合 |

### Step 4（Week 2〜3）：EAS 設定 → 広告ライブラリを実装する

`react-native-google-mobile-ads` は **Expo Go では動かない**ため、先に EAS（Expo Application Services）の設定が必要です。

```bash
# EAS CLI のインストール
npm install -g eas-cli

# EAS ログイン・プロジェクト設定
eas login
eas build:configure

# ライブラリ追加
npx expo install react-native-google-mobile-ads
```

- **完了の定義**：Development Build を実機に入れ、テスト用インタースティシャル広告が表示される
- **詰まりやすい点**：初回ビルドは EAS のサーバーでビルドされるため10〜20分かかる

### Step 5（Week 3〜4）：RevenueCat SDK で IAP を実装する

```bash
npx expo install react-native-purchases
```

- App Store Connect で「広告除去」商品（¥300）を登録してから実装する
- **完了の定義**：Sandbox テストで課金フロー（購入 → 広告消える → 復元）が動作する
- **詰まりやすい点**：Sandbox テスト用の Apple ID を別途作成する必要あり（設定 → App Store → Sandbox アカウント）

### Step 6（Week 5〜6）：TestFlight 配布 → App Store 申請

```bash
# App Store 向けビルドを作成・提出
eas build --platform ios --profile production
eas submit --platform ios
```

- スクリーンショットが必要：6.7インチ（iPhone 16 Pro Max）と 5.5インチ（iPhone 8 Plus）の2サイズ
- **完了の定義**：App Store で一般公開されダウンロードできる状態になった
- **詰まりやすい点**：審査は通常1〜3日。プライバシーポリシーURLがないとリジェクトされる
